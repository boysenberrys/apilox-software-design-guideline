Core Principles

What Apilox UI stands for

Every design decision in this system is anchored to four non-negotiable principles. When in doubt, come back to these.

Clarity First

Every element must serve a clear purpose. No decoration for its own sake. Users should never guess what something does.

Consistency Always

The same action should always look the same, across every Apilox product. Users who learn one platform should feel at home in all.

Brand Integrity

Green is Apilox's identity. It signals trust, health, and progress. Every product should feel unmistakably Apilox without being told.

Purposeful Motion

Animations inform and delight — never distract. Every transition should make the product feel alive, not laggy or flashy.

Motion & Transitions

Animation & Transition Rules

Motion should feel natural and purposeful. Use ease for most transitions. Reserve spring/bounce for delight moments. Never animate layout properties — only transform and opacity.

transition-fast
all 0.15s ease
Hover bg/color changes, icon color
transition-base
all 0.25s ease
Buttons, cards, interactive elements
transition-slow
all 0.4s ease
Modals, drawers, page transitions
hover-lift
transform: translateY(-2px)
All interactive cards and buttons
hover-lift-lg
transform: translateY(-4px)
Hero elements, feature cards
reveal-in
opacity: 0→1 + translateY(20px→0)
Scroll-triggered section entrances
modal-in
opacity: 0→1 + scale(0.97→1)
Modal and sheet open animations
CSS — Keyframes
/* Standard reveal animation for scroll-triggered sections */
@keyframes revealUp {
  from { opacity: 0; transform: translateY(24px); }
  to   { opacity: 1; transform: none; }
}
/* Modal entrance */
@keyframes modalIn {
  from { opacity: 0; transform: translateY(20px) scale(0.97); }
  to   { opacity: 1; transform: none; }
}
/* Staggered children — use animation-delay */
.card:nth-child(1) { animation-delay: 0ms; }
.card:nth-child(2) { animation-delay: 80ms; }
.card:nth-child(3) { animation-delay: 160ms; }
.card:nth-child(4) { animation-delay: 240ms; }

/* NEVER animate these — causes layout reflow */
/* ✗ width, height, top, left, margin, padding */
/* ✓ transform, opacity, filter, background-color */
Do's & Don'ts

Design Rules

These rules are non-negotiable for all Apilox products. Any deviation must be approved by the design lead.

DO — Colors
  • Use --green (#1FBC51) as the primary action color
  • Use CSS variables for every color value — never hardcode hex
  • Use green to signal success, active states, and CTAs
  • Use --ink (#0d1f14) for all headings and important text
  • Use --off-white as page background, never pure white
DON'T — Colors
  • Never use purple, orange, or pink as accent colors
  • Never hardcode #1FBC51 — always use var(--green)
  • Never use red as a primary or decorative color
  • Never use more than 3 colors in a single component
  • Never use pure black (#000000) — use --ink instead
DO — Typography
  • Use Montserrat for ALL headings and button text
  • Use Poppins for all body copy and descriptions
  • Use DM Serif Display italic only for large hero headings
  • Use JetBrains Mono for code and technical labels
  • Maintain a clear hierarchy — one h1 per page
DON'T — Typography
  • Never use Inter, Roboto, Arial, or system fonts
  • Never use DM Serif Display in non-italic or for body text
  • Never mix more than two typefaces in one section
  • Never go below 12px for any visible UI text
  • Never use all-caps for body text (labels only)
DO — Buttons & Interactions
  • Always use pill shape (--r-xxl) for buttons
  • Always apply translateY(-2px) lift on hover
  • Always use --shadow-green on primary button hover
  • Use Remix Icons exclusively — no mixing icon sets
  • Always show a green focus ring (3px, 12% opacity) on inputs
DON'T — Buttons & Interactions
  • Never use square or sharp-corner buttons
  • Never remove focus styles (accessibility requirement)
  • Never use Font Awesome or Material Icons
  • Never animate layout properties (width, height, margin)
  • Never use transitions longer than 400ms in UI interactions

Design Philosophy

The principles that guide every pixel.

Before writing a single line of UI code, understand why. These principles exist so that every Apilox product feels like one product — even if built by different teams, at different times.

The Four Principles
01
Principle One

Clarity First

Every element in our UI must communicate its purpose instantly. We remove ambiguity at every layer — from button labels to empty states. If a user has to think about what something does for more than a second, it has failed its design. Clarity means predictable patterns, honest copy, and interfaces that explain themselves without tooltips or onboarding tours.

Self-explaining UI Honest labels Zero ambiguity
02
Principle Two

Consistency Always

A user who learns Apilox Pharmacy should feel immediately at home in Apilox Clinic or Apilox HR. The same interaction patterns, the same color meanings, the same spacing rhythms. Consistency is not uniformity — it's the invisible contract we keep with users so they can focus on their work, not on learning new software.

Shared token system One component library Cross-product patterns
03
Principle Three

Brand Integrity

Green is not just a color in our palette — it is our identity. It signals trust, health, and progress. Every Apilox product should feel unmistakably Apilox at a glance, even without a logo. This means respecting our type scale, honoring our radius system, and never substituting brand values for personal aesthetic preferences.

Brand-first decisions No personal aesthetics Apilox at a glance
04
Principle Four

Purposeful Motion

Every animation we ship must earn its place. Motion should orient users, confirm actions, and signal state — not entertain. We define specific easing curves and durations (see Transitions) and apply them uniformly. A product that animates everything feels cheap. A product that animates the right things feels alive.

Informative only Defined durations Never decorative

"The best interface is one that disappears — that lets the user's task become the only thing they see."

Apilox Design Team — Internal Design Manifesto, 2024
Supporting Pillars

Healthcare Context

Our users are clinicians and healthcare workers under pressure. UI must reduce cognitive load, not add to it. Every decision should respect the stakes of the environment.

Mobile by Default

Most Apilox users access our products on phones and tablets. Design mobile-first, then scale up. Never build desktop-first and "shrink" it for mobile.

Inclusive by Design

Accessibility is not a feature — it's a baseline. Sufficient contrast ratios, keyboard navigation, and readable type sizes are non-negotiable from day one.

Performance Matters

A beautiful interface that loads slowly fails our users. Every component choice should consider bundle size, rendering cost, and perceived performance on low-end devices.

Trust Through Design

Healthcare data is sensitive. Our UI should communicate safety and professionalism. Avoid playful patterns where the domain demands seriousness and respect.

Living Documentation

This design system is never finished. As patterns evolve, update the system. Stale documentation is worse than no documentation — it creates inconsistency.

In Practice
Do
  • Use CSS variables for every color, spacing, and radius value
  • Reference this guide before building a new component
  • Test your UI at 375px wide before calling it done
  • Use the green palette to signal primary actions only
  • Update this system when you establish a new pattern
Don't
  • Hardcode hex values or arbitrary pixel values anywhere
  • Use animations for purely decorative purposes
  • Override design tokens with component-specific values
  • Design desktop-first and scale down for mobile
  • Ship a new pattern without documenting it here

Principles, not rules.

These are the beliefs that guide judgment when the rulebook runs out.
